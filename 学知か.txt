概要: サークルで11名規模（プログラマー5名、3Dモデル2名、サウンド4名）のチーム開発を経験し、プログラムリーダーを務めました。

担当領域
ゲームの核となる戦闘ロジックの実装や、ステージ上の障害物を自動生成するスクリプトを作成することで、手作業を削減し開発効率の向上に貢献しました。
ゲームロジックはゲームに対する責任が重く、完成させること優先だったため、メンバーの了承も得たため、開発経験のあった私が引き受けた。

使用技術:Unity,C#,GitHub



ゲームジャンル

3Dシューティングゲーム



ゲームコンセプト

戦闘機をカスタマイズできるシューティングゲーム



内容・役割

CGP内イベント「夏チーム」で、プログラマーリーダーとして3Dシューティングゲーム開発に携わりました。リーダーとしてメンバーのタスク割り振り、進捗管理、コードのマージ、unityroomへのリリースを担当し、プログラマーとしてゲームの核となる戦闘ロジックの実装、Particle Systemでのビーム作成、機体や障害物の移動制御、スコア保存など、主に戦闘画面に関わる機能開発を全て担当しました。

制作終盤で一部メンバーと連絡が取れなくなるトラブルが発生しましたが、手の空いているメンバーに適切に指示を出し、作業を分担することで、目標の締め切りまでにゲームを完成させることができました。この経験を通じて、スケジュール管理の重要性や、リーダーとして的確な指示を出す必要性を改めて実感しました。このゲームはゲームクリエイター甲子園2023にも提出しました。

1. タスクの割り振りは、画面ごとに行いました。例えばタイトル画面、メニュー画面、ショップ画面、戦闘画面などです。こうすることで、他者のスクリプトの変更によるコンフリクトを予防しました。また、進捗管理は週に1度3DやSE班も含めた定例会議を用意し、そこで進捗報告を行っていました。また、プログラマのみのチャンネルも作成し、全体で聞きづらいことなどを質問できる場も用意しました。

2. 企画が終了し、いよいよ実装に取りかかるタイミングでメンバーの一人が音信不通になってしまいました。そこで、その方に割り振っていたタスクを誰かが担当する必要が生まれました。多くのメンバーは未経験者であったこともあり、追加タスクを与えるべきではないと判断し、基本は私が巻き取りつつ、もし他の方が早めにタスクを終えることができたなら、依頼しようと考えていましたが、皆タスクを早めに終わらせ、自主的に余剰タスクを引き受けることを提案していただけたため、余裕を持って作業を終わらせ、最終発表会までにリリースまで完了させられた。

3. パーツを購入し自分好みにカスタマイズできるというのをウリにしていたので、パーツによって変わるステータスや打てるビームの効果を考えるのに苦労しました。あるものには回復をつけたり、あるものには一撃を重くしたり、またある機体にはスピードを、別の機体には耐久力を与えたりしたのですが、結局連射性能の高い機体で連射することで、一撃が高い機体や攻撃力が高い機体のメリットを亡き者にしてしまったことは、バランス調整の失敗として、身にしみています。

4. 私が担当したのは戦闘画面ですが、特にステージ配置を工夫しました。3D班から仕上がってきていたステージをかたどる惑星などを最初は手で1つずつ座標を指定しておこうと思っていたのですが、プログラムで簡略化できないかと考え、ランダムで惑星を着たいが通る範囲に設置するスクリプトを作成して、挑戦するたびに障害物の位置が変わるステージを作成できました。このことで開発スピードが上がり、中間報告では、もう戦闘シーンのひな形は完成させていました。

以下は事実と異なるエピソードの例と実際の内容です。

＞＞また、エンジニア以外のメンバーも効率的に作業できるよう、Unityエディタの拡張機能を自作しました。具体的には、配置ルールが複雑なステージ上の障害物を、パラメータ調整だけで自動生成・整列させるスクリプトです。これにより、レベルデザイナーがプログラムを介さず試行錯誤できる環境を構築し、チーム全体のアセット制作スピードを飛躍的に向上させました。

ここは、やったこととしてはそうですが、レベルデザインを担当できるプランナーなどはチームにおらず、実質的にプログラマーが行うことになっていたので、そういった理由ではないです。単に、ステージに障害物を手で置いていくのがめんどくさかったのと、ステージに挑戦したときに、ゲーム性が少しは変化がつくことを期待してです。エディタ拡張ではなく、単にプレハブを一定間隔で、画面に見える位置にランダムに作り出すだけです。

＞＞常に自分のタスクに余裕を持たせていたため、即座にフォローへ回り、クオリティを維持したまま期限内に作品を提出しました。結果として一次審査を通過し、チーム一丸となって「一つのプロダクトを完成させる」という大きな成果を得ることができました。

私が一番しんどいところを引き受けていたのでタスクに余裕は持たせていません。ただ、飛ばれる可能性が非常に高そうな方がいらっしゃったので、何か起きても大丈夫なようにその方のタスクを変えが聞く(最悪機能として消せる)ようなところにしていたのが実情です。

また、ゲームクリエイター甲子園は、作品を提出したらあとは特にすることがなく、ということで、別にその後の連絡も何もなかったので、書くべきかは疑問です。

＞＞開発初期、11名という人数の多さから、同じシーンファイルを同時に編集することによるマージミスや、機能間の依存関係が強すぎて一箇所の修正が全体に波及する不具合が頻発しました。プログラムリーダーとして、このままでは後半のバグ修正が追いつかなくなるという危機感を抱きました。

そんな不具合は怒っていません。そもそもそんなのが起きないようにするためにこうしているのです。

＞＞コードレビューを通じて「誰が読んでも意図が伝わる、疎結合なコード」への修正を促しました。

ここまではしていません。初心者にするものでもないでしょうし、私も経験あるとはいえそこまでだったので、単にそれぞれが自身の名前のブランチを切り、それぞれのシーンで、それぞれのフォルダ内でのみ作業を行う形にしていましたね。マージも最後の結合のみです。つなぎ合わせは少し苦労しましたが。

＞＞第二に、共通基盤となる「ゲームマネージャー」の再設計です。各機能が互いの内部実装を知らなくても情報をやり取りできるよう、イベント駆動型の設計（Observerパターン等）を一部導入しました。これにより、一人の担当者が機能を追加しても他の箇所に影響を与えない、堅牢なシステムを構築しました。

初心者同士のイベント(特にプログラミングが初めてのパターンもいるかも)なので、設計なんかしていません。自由にさせています。情報のやり取りは原始的ですが、PlyerPrefsに頼りましたね。どのキーで登録しているかを共有さえしていれば、何とかなりましたので。皆さんからしてもわかりやすかったそうです。

## 開発エピソード4(ゲーム)(個人開発)(Unity)

ハッカソンでの失敗を教訓とした3D脱出ゲームのリベンジ開発です。以前、複雑なステージの忠実な再現を目指した結果、動的生成の実装が不可能となり、ゲームを完成できませんでした。この悔しさから、1週間で企画からリリースまで全て行う個人開発に夢中になりました。

この開発で最も熱中したのは、前回失敗した動的生成の実装を簡略化する仕組みを考案することです。

ステージの段差や曲がり角が実装を複雑化させていたと分析し、ステージ構成、ユーザーの入力を根本から見直し、大幅な変更を加えました。これにより、ステージの連続生成を簡潔に実装できるようになりました。

この創意工夫により、動的生成の実装難易度を大幅に下げることができ、開発期間内に余裕を持ってゲームを完成させ、リリースに至りました。

この経験を通じて、失敗をネガティブに捉えず、解決すべき課題として捉え直すプロセスに心から夢中になれることを再認識しました。この課題解決への情熱こそが、私の開発の原動力となっています。

[開発背景]: 個人開発(ハッカソンでの反省点を活かした再挑戦)

期間: 2024年3月10日～3月17日（1週間）

使用技術: Unity, C#, Mixamo, GitHub Desktop (GitHub), Visual Studio Code



ゲームジャンル: 3D脱出ゲーム



内容・役割

前回のKC3HACKでゲームを完成できなかった悔しさから、リベンジとして、同じく「8番出口」をオマージュした脱出ゲームを個人で制作しました。企画からステージ作成、テクスチャ、設計、実装、テスト、リリースまで全て一人で担当しました。

前回の失敗原因が複雑なステージ設計による動的生成の不具合だと考え、今回は異変の判定を分かれ道で実装し、z軸の調整のみでステージが生成できる仕組みを考案することで、動的生成の実装を大幅に簡略化できました。NPCの歩行アニメーションなど、以前の進捗も効果的に流用しながら、結果として開発期間内に余裕を持ってゲームを完成させ、リリースすることができました。この経験を通じて、失敗から学び、具体的な課題解決へと繋げる実践力と、個人プロジェクトの完遂能力を養いました。

1. KC3HACKでの失敗は多重ネストによるバグではありません。実際の失敗の原因は「8番出口」を詳細に表現しようとしたことが原因です。「8番出口」というゲームは、とある道を動的生成で永久的にループさせ、異変があれば戻り、異変がなければ進んでいき8回連続で正解できるとゲームクリアとなるゲームです。本家では、これらを実際のモデルを参考にしつつもループしやすいステージ設計をされていたのですが、当時の私たちは関西在住なこともあり、モデルの8番出口について調べていなかったため、実際の地下道をモデリングしているのだと勘違いしていました。なので、実際にステージ設計を行う際も、現地に出向き、なるべく8番出口に近い対象の地下通路を忠実に再現しようとしました。その結果、階段とスロープによる段差、および、コーナーと扉が追加されたことで、単なるステージ単位の動的生成ではステージが連続できなくなっていることが発覚し、1週間で作り上げることは不可能となってしまいました。また、動的生成に取り組んだことのあったメンバーもいなかったため、余計に実装難易度が上がったことも原因です。

2. そこで、今回の個人開発では、実際の地下通路等は参考にせず、架空の地下通路で作成することにしました。架空の地下通路にすることによって、動的生成を簡単にできるようにステージそのものを改良できるようになりました。それが、「z軸の調整のみでステージが生成できる仕組み」です。先ほど申し上げたように本家では異変がなければ前へ進む、異変があれば戻ることがユーザーの入力になります。これを忠実に再現する場合、ステージはプレイヤーが存在する現在位置の前と後ろの両方に作成する必要があります。また、本家では曲がり角を実装されているため、これも実装すると、単なるz座標の移動だけでは連続したステージができません。そこで、ユーザーの入力を前進後進ではなく、左右の分かれ道に進んでもらうことで、本家の曲がり角の意図の一つである、ステージ生成の判定と実行を行うための時間を用意しつつ、分かれた道をもう一度一つの道に合流させることで、元のただまっすぐな道に戻せるため、動的生成をz軸の変更だけで行えるようにしました。

3. これらの工夫はすべて実装を簡略化するために行いました。当初企画の段階で現実に忠実なステージを使って作成しようとしていたとき、本当に大丈夫なのかと不安になりましたが、事前に役割分担をした際はゲームにおいて重要なNPCや異変の素材の準備を担当することになっていました。そのため、実際に開発する人がやりたいなら、また、私よりも開発歴も長い人だったので、私の心配など不要だと考え、口に出さなかったのが私の失敗だったと考えています。特に開発時間が短いときは完璧を狙わずに、本当に必要なコアな機能から完成させ、その後徐々に複雑化させていくことが大事だと学びました。これは先のunity1weekに提出した育成シミュレーションゲーム開発時にも活かされています。

1. 「議論」をシステム化する際の壁について

もう当時でもすでにAIはフリーの議論であれば言い負かしてくるような議論を行えていました。ですので、あなたのいう、AIが同じ事を何回も言う、議論がかみ合わないという課題は最初からなかったですね。しかし、今回対象としたディベートは議論のフォーマットが定まっていますので、それに従うように指示を出すのが、非常に頭を悩ませましたね。結局、各議論のセクション(立論などの)と立場に合わせたプロンプトをそれぞれ用意し、それまでの議論の履歴と合わせて使用する力技で何とかしてしまいました。プロンプトでは文字数も指定していましたが、あまり守られてはいなさそうでしたね。プロンプトの文章が多いので、すべてを守らせることは難しいのはわかっていますし、あくまでトークンでのカウントなので文字数とは数え方が違うのもわかりますし。

2. 「判定（ジャッジ）」の納得感について

ディベート用のプロンプトを作成する際に、ディベートの例えば立論はこうやって組み立てるのですよ、といったアドバイスを含めていました。これは、ディベートの公式団体(日本)が出している参考記事を用いています。ですので、AIの判定にもこれと同じ基準を用いました。ですので、理論上はAIは毎回100点の回答を出せるはずです。当然そうなりませんが。

判定が納得がいかないは実際ありましたが、これは現実のディベートでもそうなので、割り切ってしまいましたね。結局人によって判断が変わってしまうのがディベートの不確実で面白いところでもあるので、ですので、判断基準はいかに論理的に説明しているか、必要な情報をきちんと伝えられているかといった、テクニック重視で、ディベートの勝敗の影響度はおまけ程度にしていました。

3. 実装上の「泥臭い」苦労について

1番は、やはり、1.でもあげたディベートセクションごとのプロンプト渡しですね。当初は楽にできないかなと考えたのですが、思いつかなかったので、力業で実装したことですかね。学習者の立場によって、AIの行うセクションの順番やプロンプトの送信タイミングが変わってしまうので、すべてで場合分けをし、それぞれ実装するスタイルになってしまいました。また、このような力業の実装をしたにもかかわらず、のちの想定外で詳しく話しますが、仕様変更が発生したのも痛かったです。ですので、修正範囲がでかくなったのもしんどかったですね。

4. 評価実験での「想定外」について

1番の想定外は私は何度もAIとディベートをしていたこと、および、そもそもの思考が人より速めということで、あまり時間がかからず実験を終えられていたので、大丈夫だと考えていたのですが、想定よりもはるかに長い時間ディベートに時間がかかっているということに、本実験前の仮実験で気づき、本番でこれはまずいと考え、即座にディベートのセクション数の削減や、AIが提示する根拠の数の修正に取り掛かりましたね。基本的に論点が多い方がすべてに回答しなくては行けず、文章が長くなりがちなので。これは、初心者に時間制限を設けては思うように議論が展開できないまま終わってしまうと考え、制限時間を設けなかったのが裏目に出て、まじめに長時間熟考してしまう方が多かったのは読めなかったですね。

結論


  このプロジェクトは、単なるAIとのチャットアプリケーションではなく、「ディベート（討論）」という明確なルール
  と目的に沿ってユーザーの論理的思考力や表現力を体系的に評価・フィードバックする、非常に意欲的なシステムです
  。特に、ディベートの各フェーズ（立論、反対尋問、反駁など）ごとに評価基準を変え、AI（GPT）を審判役として利
  用して多角的な採点を行う仕組みは、このプロジェクト最大のこだわりであり、優れた点です。

  ---

  優れている点・こだわりが感じられる点（詳細）

  1. 独自性の高い「ディベート評価システム」の実装


   * こだわり: debate_logic.py と scoring.py は、このプロジェクトのために作られた完全に独自のロジックです。こ
     れはDjangoの標準機能では全く提供されません。一般的なチャットボット開発とは一線を画し、「ディベートのトレ
     ーニング」という明確な目標達成のために、複雑なロジックが組まれています。
   * 優れている点:
       * 体系的なディベートフロー: debate_logic.py の handle_turn 関数では、ディベートのターン（turn_count）と
         ユーザーの立場（user_position）に応じて、現在のフェーズ（立論、反対尋問など）を厳密に管理しています。
         これにより、ユーザーは実際のディベートに近い形式で対話を進めることができます。
       * フェーズごとの重み付け: 各フェーズの評価結果に score * 18 / 100 や score * 23 / 100
         のような異なる重みを付けてスコアを算出しています。これは、ディベートの各パートの重要度を考慮した本格
         的な採点ルールを模倣しようという、強いこだわりを感じさせます。


  2. AI（GPT）を「審判」として活用する高度な仕組み


   * こだわり: このプロジェクトでは、AIを単なる対話相手としてだけでなく、ユーザーの発言を評価する「審判（ジャ
     ッジ）」として活用しています。scoring.py の evaluate_argument 関数がその中核です。
   * 優れている点:
       * 状況に応じた評価プロンプトの動的生成: scoring.py では、ディベートのフェーズ (phase)
         ごとに、settings.py に定義された異なるシステムプロンプト（PR_J_MAIN,
         PR_J_QUESTIONなど）と、それまでの対話履歴を組み合わせて、評価用のプロンプトを動的に生成しています。こ
         れにより、各フェーズの目的に沿った的確な評価をAIに依頼することが可能になっています。
       * 詳細な評価項目とフィードバック: parse_evaluation_response 関数は、AIからの評価テキスト（例：「1: 5点,
          2: 4点...」）を正規表現でパースし、項目ごとの詳細なスコア（detailed_scores）と、文章によるフィードバ
         ックを抽出しています。単なる合計点だけでなく、どの観点が優れていたか・改善すべきかをユーザーに提示で
         きるため、教育的価値が非常に高いです。

  3. Djangoの機能を活用した堅実なバックエンド設計


   * こだわり: 複雑なディベートロジックを、Djangoのセッション管理機能を使ってステートフル（状態を保持する）に
     実現しています。
   * 優れている点（Djangoの機能）:
       * セッション管理: request.session
         を巧みに利用して、ディベートの対話履歴（conversation）、各ターンのスコア（scores）、フィードバック（f
         eedbacks）などをユーザーごとにサーバー側で保持しています。これにより、複数ターンにわたる複雑なやり取
         りを、状態を失うことなく管理できています。これはDjangoのセッション機能がなければ実現が難しい部分です
         。
       * 設定の分離: settings.py に PR_J_MAIN のようなプロンプト定義を分離して記述している点も、コードの可読性
         とメンテナンス性を高める良い設計です（これはDjangoの標準的な使い方です）。

## 個人開発(ゲーム)(C++)(Dxlib)

本作品は C++ / DxLib を用いて制作した、 ゲームに必要な数学（ベクトル・距離・円衝突判定）の理解を目的としたミニゲームです。

「数学・物理の知識が、実際のゲームロジックとしてどのように使われるのか」を 最小構成のゲームとして体験・説明できることを目標に制作しました。

ゲーム内容
プレイヤーは 円 として扱われます
WASDキーで移動します
開始から 2秒後にプレイヤーは非表示 になります
障害物（固定円）や画面外に触れると ゲームオーバー
ゴール円に到達すると ゲームクリア
クリアまでの時間をスコアとして表示します
視覚情報に頼れない状態でも、 距離感・位置関係・判定を意識した操作が求められる設計です。

操作方法
キー	動作
W / A / S / D	プレイヤー移動
SPACE	ゲーム開始
ENTER	タイトルに戻る
使用技術
言語：C++
ライブラリ：DxLib
開発環境：Visual Studio 2022
実装した主な数学・ゲーム技術
1. 2Dベクトル（Vec2）
加算・減算
長さ（ノルム）
距離計算の基礎
位置差ベクトル = A.pos - B.pos
2. 円同士の衝突判定
円同士の当たり判定は、 中心間の距離が半径の合計以下かどうかで判定しています。

|A - B| ≤ rA + rB
平方根計算を避けるため、距離²で比較しています。

3. 障害物管理（vector）
障害物は vector<Circle> で管理
描画・判定をループで共通化
コースは規則的に配置
for each obstacle:
  if Hit(player, obstacle): GameOver
4. ゲームステート管理
enum class GameState
{
    Title,
    Playing,
    GameOver,
    Clear
};
状態に応じて入力・更新・描画を制御
不正な入力を防止
5. 画面外判定
プレイヤーが画面外に出た場合、 非表示状態では復帰が困難になるため、 即ゲームオーバーとしています。

0 ≤ x ± r ≤ ScreenWidth
0 ≤ y ± r ≤ ScreenHeight
設計上の工夫・意図
プレイヤー・敵・ゴール・障害物を すべて円として統一
判定ロジックを完全に共通化
プレイヤー非表示により、 数学的な距離感覚をゲーム体験に直結
再利用可能な数学・判定処理は ヘッダ・cppに分離
ファイル構成
project1/
 ├─ main.cpp
 ├─ Math.h / Math.cpp
 ├─ Collision.h / Collision.cpp
Math：ベクトル計算
Collision：円衝突判定・画面外判定

本リポジトリをクローン
.sln を開いてビルド
